Methodology

How we source, verify and publish.

Nirman Darpan is a civic data product. This page is a public record of how every number, headline and accountability flag on the site got there.

Project records

Each project's core record — name, location, mandate level (Centre/State/District), category, status, budget, contractor, accountable leads, milestones — is curated from public sources. The current sources we draw from are catalogued at data/sources.md. They include:

  • Central agencies: NHAI, MoRTH, BRO, AAI, Indian Railways / RVNL, SJVN, HPPCL, MoHFW (PMSSY), Jal Shakti, MoHUA Smart Cities, BBMB.
  • State authorities: HP State Portal, HP PWD, HP Jal Shakti Vibhag (IPH), HPSDMA, SJPNL, HP Education Department, HP Tourism Department, HPRTC.
  • District-level: DC offices, Municipal Councils, PMGSY OMMAS.
  • Press (for cross-verification): The Tribune, Hindustan Times, The Indian Express, The Hindu, Divya Himachal, Amar Ujala.

Project records are hand-curated. Some figures — particularly budget and spent values for in-progress projects — are best-available public estimates and refined as audits and progress reports become available. The git commit history shows every change to every record, making the dataset fully auditable.

How the progress percentage works

The completion percentage on each project is a reported figure, not a computed one. It is set editorially from what public sources say — departmental progress reports, press coverage of milestone completions, official statements — and each project shows the date the figure was last reviewed ("as of …"). Where a specific source backs the number, it is linked next to the figure. We do not survey sites ourselves, and we do not derive the percentage from budget-spent ratios (money spent is a poor proxy for physical progress on Indian infrastructure projects, where land acquisition can consume large sums before construction starts). If you believe a figure is stale or wrong, use the corrections process with a source and we will review it within 7 days.

Press mentions ("Public conversation" panel)

Each project panel includes a count of recent press mentions and links to the top headlines. This is aggregated from Google News (India) in English (en-IN) and Hindi (hi-IN) locales using each project's keyword set. The aggregation:

  • Queries phrases (not loose keywords) to reduce false positives.
  • Deduplicates by URL stem.
  • Stores per-project snapshots in data/social/<id>.json in the repo — fully visible.
  • Refreshes via GitHub Actions on Mondays and Thursdays.

We link to headlines; we never repost the article text. This respects publisher copyright and avoids ToS issues. We also do no automated sentiment scoring on press mentions — reading the headline is a more honest signal than collapsing it into a percentage. See "What we don't do" below for why.

Citizen voices (Public perception panel)

Verified residents can sign in (passwordless email link) and cast one vote per project — helpful or concerns. The "one vote per resident per project" rule is enforced by a database-level unique constraint, not just a UI nudge.

Comments require the same sign-in. Both follow the code of conduct.

Sample-size threshold: we do not show vote percentages until at least 25 verified votes exist on a project. Below that, percentages from a tiny sample are statistical theatre — we show only counts.

Accountability records — the editorial chain

Each project's Accountability panel tracks five categories: incidents, defects, audits, grievances, and litigation. Three things can put an item into the queue for editorial review:

  1. The press sweep (above) flagged it via accountability keywords.
  2. A verified resident submitted it through the project page, with a source URL.
  3. Editorial staff entered it directly while reading press / official records.

None of the three appear on the public site until an editor reviews and verifies them against a primary source. The accountability panel that visitors see contains only items that have passed that check.

Editorial decisions (approvals, rejections, edits) are logged on the corrections page. The current editor is the project owner; widening to a small moderator team is in the roadmap (see docs/ROADMAP.md, Phase 4).

Auto-sync

A scheduled job runs Mondays and Thursdays at 19:00 UTC (≈ 00:30 IST). It refreshes:

  • data/meta.json — the "Auto-synced" timestamp at the top of the site.
  • data/social/*.json — press snapshots per project.
  • sitemap.xml — search engine index.

The workflow definition is at .github/workflows/sync.yml.

What we don't do (and why)

  • No automated sentiment scoring on press headlines. AI models can mislabel an "audit uncovers Rs 22 cr scam" headline as neutral because it's factual. A reader who clicks the link gets a more accurate impression in two seconds than any model can summarise.
  • No anonymous comments. Every Indian civic comment thread without auth gets flooded by vendor PR campaigns and partisan astroturf within weeks. Verification is friction, but it's the only way the signal stays meaningful.
  • No reposting of press text. Headlines + links only.
  • No tracking pixels or analytics. See privacy.
  • No blended scores. Verified-resident votes and press signal are different qualities of data. Showing one number that combines them would hide where the actual conviction lives. They are presented separately.

Limitations

  • Budget and spent figures for active projects are best-available public estimates. They are refined as audits emerge.
  • Site coordinates are approximate (corridor midpoint for multi-district works); refined when field-verified.
  • HP-only for now. The schema is portable; replication to another state is welcome.
  • Coverage skews to projects above ~₹100 crore. Smaller works fall through the cracks; we'd like to fix that with citizen submissions.

If you spot an error, see corrections for how to report it.
